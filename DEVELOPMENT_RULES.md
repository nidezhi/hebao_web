# DZCOM 前端开发铁律

## 信息架构

1. 一个页面只承担一个明确的业务职责。配置、记录、资讯、分析等不同工作流必须拆成独立路由。
2. 同一领域存在三个及以上页面时，必须使用父子菜单组织，不得把所有功能堆在一个页面或一级菜单中。
3. 工作台用于业务概览、异常识别和趋势判断，不得作为快捷链接集合或功能说明页。
4. 页面间跳转必须有明确业务必要性。详情、编辑和状态变更优先在当前页面通过抽屉、弹窗或局部面板完成。
5. 兼容旧路径时使用路由重定向，不保留重复页面。

## 页面结构

1. 列表页默认结构为“紧凑筛选区 + 主操作 + 数据表格”，主操作每页原则上不超过一个。
2. 筛选项只保留高频字段，低频条件放入高级筛选，不得让工具栏占据主要视区。
3. 详情页必须将结构化数据转换为指标、描述列表、图表或业务区块，不得直接堆放原始 JSON。
4. 一个页面不得同时出现多组互不相关的 tab。不同工作流应拆路由，同一对象的紧密详情可使用 tab。
5. 页面标题、菜单选中态和父级上下文必须与当前路由一致。

## 工作台与图表

1. 工作台图表必须使用真实接口数据，禁止用静态演示数据冒充业务数据。
2. 图表必须有明确标题、单位、图例和空数据状态，并随容器尺寸响应式调整。
3. 图表类型应匹配数据：结构占比使用饼图，时间变化使用折线图，分类比较使用柱状图。
4. 图表依赖必须按需引入，避免全量打包。

## 交互与视觉

1. 管理后台以扫描效率和重复操作效率优先，避免营销式大标题、装饰性卡片和无意义留白。
2. 页面区块不使用卡片嵌套；卡片圆角不超过 8px。
3. 操作按钮按频率和风险排序：主操作突出，次要操作收敛到菜单，危险操作必须二次确认。
4. 状态统一使用标签或开关；类型和模式使用选择器或分段控制；图标优先使用现有图标库。
5. 桌面端和移动端均不得出现文字、按钮、表格或图表重叠。

## 工程约束

1. 新增页面前先检查现有路由、API 封装、类型和组件模式，优先复用现有约定。
2. 接口字段必须在 TypeScript 类型中同步体现，JSON 字符串必须通过结构化解析处理。
3. 完成修改后必须执行 `npm run type-check` 和 `npm run build`。
4. 涉及主要页面结构或图表变更时，必须启动本地服务检查目标路由。
