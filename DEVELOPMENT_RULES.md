# DZCOM 前端开发铁律

当前阶段：**按稳定 API 文档生成业务页面期**。登录、鉴权、路由、布局、UI 规范和 API 基础设施保留；业务页面必须严格按 `mock/18-frontend-generation-blueprint.md` 生成，`mock/api.md`、`mock/13-frontend-refactor-plan.md` 作为 review 条件。

## 0. 当前阶段边界

1. 保留登录页、auth store、路由守卫、httpClient、endpoints、基础布局、UI 规范页、API 文档舱和开发铁律页。
2. 业务页面必须调用真实 API client，不能继续展示统一重置占位。
3. API client、DTO、adapter、dictionary 是页面生成的前置契约；页面不得直接解析复杂后端结构作为主要逻辑。
4. 禁止根据旧页面效果修补；业务页必须按稳定 API 文档和 18 号蓝图重新生成。
5. 恢复业务页面时，必须先补齐类型、接口、字典和 adapter，再生成 pages。

## 0.1 业务生成铁律

1. 所有业务路由必须展示真实工作台、列表、详情、图表或明确空状态。
2. 业务页面必须通过 `entities/*/api.ts` 调用接口，不能在页面组件直接写接口路径。
3. 禁止静态演示数据、临时图表数据、假聚合数据。
4. 登录功能必须可用；未登录访问受保护路由仍必须跳转登录页。
5. 生成完成后必须执行 type-check、build，并至少抽查登录页和一个受保护业务路由。

## 1. 信息架构

1. 一个页面只承担一个明确职责。配置、记录、资讯、分析等不同工作流必须拆成独立路由。
2. 同一领域存在三个及以上页面时，必须使用父子菜单或工作区分组组织。
3. 工作台用于业务概览、异常识别和趋势判断，不得作为快捷链接集合或功能说明页。
4. 页面间跳转必须有明确业务必要性。详情、编辑和状态变更优先在当前页面通过抽屉、弹窗或局部面板完成。
5. 兼容旧路径时使用路由重定向，不保留重复页面。

## 2. API 与类型

1. 业务接口统一通过 API client 调用，页面组件不得直接散写请求路径。
2. 接口路径集中维护；同一接口不得在多个文件重复声明。
3. 所有接口默认使用 `POST + JSON body`，除非后端 API 文档明确声明例外。
4. DTO、领域 ViewModel、表单 Model 必须分层，不在页面里临时拼接复杂后端结构。
5. 字典集中管理，筛选项、下拉、Tag、图例、文案共用同一份定义。
6. 图表数据必须在 adapter 层转换，组件只消费稳定展示模型。
7. JSON 字符串必须结构化解析，解析失败必须有明确降级状态。

## 3. 页面结构

1. 列表页默认结构为“紧凑筛选区 + 主操作 + 数据表格”，主操作每页原则上不超过一个。
2. 筛选项只保留高频字段，低频条件放入高级筛选，不得让工具栏占据主要视区。
3. 详情页必须将结构化数据转换为指标、描述列表、图表或业务区块，不得直接堆放原始 JSON。
4. 一个页面不得同时出现多组互不相关的 tab。不同工作流应拆路由，同一对象的紧密详情可使用 tab。
5. 页面标题、菜单选中态和父级上下文必须与当前路由一致。

## 4. 操作舱 UI 规范

1. UI 方向为“科幻操作舱”：深色底、网格线、霓虹状态、玻璃面板、高信息密度。
2. 科幻感必须服务操作效率，不得牺牲可读性、响应式布局和数据扫描效率。
3. 第一屏必须是实际工作台或明确等待态，不做空泛欢迎页、营销页或说明书页。
4. 页面区块使用全宽工作区或分栏布局，不做漂浮装饰区块。
5. 卡片只用于独立对象或清晰区域；不做卡片套卡片；圆角不超过 8px。
6. 状态必须有明确颜色和文案：正常、运行中、降级、待复核、拒绝、失败、无权限、数据不足。
7. 按钮按频率和风险排序：主操作突出，次要操作收敛，危险操作必须二次确认。
8. 桌面端和移动端均不得出现文字、按钮、表格或图表重叠。

## 5. 工作台与图表

1. 工作台图表必须使用真实接口数据，禁止用静态演示数据冒充业务数据。
2. API 未就绪时展示 API Standby 或空状态，不展示假图表。
3. 图表必须有明确标题、单位、图例和空数据状态，并随容器尺寸响应式调整。
4. 图表类型应匹配数据：结构占比使用饼图，时间变化使用折线图，分类比较使用柱状图。
5. 图表依赖必须按需引入，避免全量打包。

## 6. 错误与权限

1. 401、403、404、409、500 必须映射为清晰前端状态。
2. 403 统一展示为“当前记录不可见或已无权限访问”，不得暴露内部权限细节。
3. 数据不足、接口缺失、字段缺失、解析失败必须在 UI 中显式展示，不得静默失败。
4. 所有高风险动作必须有二次确认、失败原因和可追踪上下文。

## 7. 工程约束

1. 新增页面前先检查现有路由、API 封装、类型和组件模式，优先复用现有约定。
2. 完成修改后必须执行 `npm run type-check` 和 `npm run build`；若环境无 npm，使用等价命令并记录原因。
3. 涉及主要页面结构或图表变更时，必须启动本地服务检查目标路由。
4. 不得提交构建产物、临时日志、调试脚本或未使用代码。
5. 删除业务代码时，必须确保路由、导入、类型引用同步清理。

## 8. 后续 API 文档生成后的重建顺序

1. 解析 API 文档和后端提示词产物，按业务域识别接口、请求体、响应体、状态码和业务约束。
2. 生成或更新 `shared/api/httpClient`、`shared/api/endpoints`、错误处理和 Result 解包。
3. 生成 `entities/*/api`、`entities/*/model`、adapter、dictionary；字段不明确时先停，不进入页面生成。
4. 生成 `features/*` 表单与动作流；高风险动作必须带二次确认和失败状态。
5. 生成 `pages/*` 工作台、列表、详情、抽屉、图表和空状态。
6. 执行 type-check、build、本地路由冒烟；失败则回滚或修复，不进入下一阶段。
