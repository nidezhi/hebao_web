# DZCOM 开发规范基线

本文档是 DZCOM 项目的开发规范权威基线，用于指导当前仓库后续所有设计、编码、重构、评审与 AI 辅助开发活动。

## 1. 适用范围

- 适用于 `dzcom` 仓库的全部新增业务模块、数据库变更、接口设计与架构调整。
- 本文档是新的工程基线；历史示例代码不再作为实现参考。
- 规范、需求、架构、数据库和交付节奏发生变化时，必须同步更新 `docs_new`。

## 2. 核心原则

- 领域优先：先定义业务边界、术语和规则，再落代码和表结构。
- 分层清晰：业务规则留在领域层，编排留在应用层，传输协议留在接口层，技术实现留在基础设施层。
- 小步交付：每次只推进一个有界上下文内的一条能力链路，避免“大而全”的一次性交付。
- 安全前置：密钥、凭据、风控、审计和敏感数据处理优先于功能便利性。
- 可测试与可观测：关键规则必须可验证，关键流程必须可追踪。
- AI 最小信任：AI 只辅助提效，不替代设计判断、代码审查和测试验收。

## 3. 分层职责

### 3.1 领域层

- 定义聚合、实体、值对象、领域服务、领域事件、仓储接口。
- 负责不依赖传输协议和 ORM 的核心业务规则。
- 不直接依赖 Controller、数据库框架和第三方 SDK。

### 3.2 应用层

- 负责编排用例、控制事务、做对象转换、调用领域对象和仓储接口。
- 不承载核心业务规则，不直接写 HTTP 协议细节。
- 对外暴露 use case 级服务，而不是面向表结构的 CRUD 拼装。

### 3.3 接口层

- 提供 REST API、请求响应对象、OpenAPI 注解和参数校验。
- 只做协议适配、输入校验和结果返回。
- 不在 Controller 中写跨对象业务逻辑。

### 3.4 基础设施层

- 提供仓储实现、数据库访问、缓存、消息、外部服务集成、定时任务等技术能力。
- 可以依赖 MyBatis、Redis、HTTP Client、消息中间件等技术框架。
- 不能把数据库访问细节反向泄露到领域模型。

## 4. 包结构要求

- 第一层必须体现 DDD 领域层级：`domain`、`application`、`infrastructure`、`interfaces`、`common`。
- 领域层级内部按结构职责拆分，结构职责内部再按业务域拆分。
- 示例：`domain/model/account` 表示领域模型结构下的账户业务，不能简化为顶层 `model/account`。
- 禁止以业务域作为第一层复制整套架构，也禁止丢失领域层级直接建立顶层 `entity`、`service`、`controller`。

推荐结构：

```text
src/main/java/com/example/dzcom/
├── domain/
│   ├── model/<context>/
│   ├── repository/<context>/
│   ├── service/<context>/
│   ├── event/<context>/
│   └── enums/<context>/
├── application/
│   ├── service/<context>/
│   ├── command/<context>/
│   ├── query/<context>/
│   ├── dto/<context>/
│   └── assembler/<context>/
├── infrastructure/
│   ├── persistence/entity/<context>/
│   ├── persistence/repository/<context>/
│   ├── security/<context>/
│   ├── session/<context>/
│   └── config/
├── interfaces/
│   ├── controller/<context>/
│   ├── request/<context>/
│   └── response/<context>/
└── common/
```

`common` 仅保留真正跨领域复用的异常、响应、分页、常量和稳定能力契约。

## 5. 编码要求

- 使用明确、完整的业务命名，避免无语义缩写。
- 类、公共方法、关键字段、业务规则和复杂流程必须有详尽中文注释；注释说明职责、约束、边界或原因，禁止只翻译代码。
- 方法 JavaDoc 属于开发规范铁律，所有新增或修改的方法必须按照以下格式编写；`@param`、`@return` 和 `@throws`
  必须与真实签名和行为一致，不适用的标签不得虚构，`@author` 固定为 `dz`，`@date` 使用 `yyyy-MM-dd` 格式：

```java
/**
 * 根据用户ID查询用户详细信息。
 *
 * @param userId 用户的唯一标识索引
 * @return 返回包含用户详细信息的 UsrInfo 对象
 * @throws IllegalArgumentException 当 userId 为 null 或负数时抛出
 * @author dz
 * @date 2026-06-14
 */
```

- 单个方法保持单一职责，复杂逻辑拆成清晰的私有方法或领域行为。
- 值对象优先不可变，在构造时完成校验。
- 领域对象优先表达行为，不做贫血数据模型。
- Entity、领域对象、Command、DTO、Request 和 Response 优先提供 Builder，业务代码避免直接调用长参数构造器。
- 集合的筛选、转换、分组、去重和汇总优先使用 Stream；涉及副作用、事务写入顺序、异常短路或可读性下降时使用命令式代码。
- 日志记录关键状态变更、失败原因和关联 ID，禁止输出密码、Token、密钥等敏感数据。
- 配置项一律外置，仓库中不保留明文生产凭据。
- 默认持久化方案为 MyBatis Mapper XML + Flyway；每个持久化实体必须具有独立 Mapper Java 接口和同名 XML。
- 禁止使用 JPA、EntityManager、JPQL、QueryWrapper、BaseMapper 内置 CRUD 或在 Java 注解中编写 SQL。

## 6. API 与安全要求

- 所有接口统一返回 `Result<T>`。
- HTTP 接口统一使用 `POST`，接口路径使用固定的用例动作名称，禁止使用 `GET`、`PUT`、`PATCH` 或 `DELETE`。
- 所有业务请求参数必须封装为请求对象并通过 `@RequestBody` 接收，禁止使用 `@RequestParam` 和
  `@PathVariable` 在 URL 中传递业务参数；确无业务请求参数的接口可以不声明请求体。
- 请求参数使用 `@Valid` 或 `@Validated`，校验失败统一交给全局异常处理。
- OpenAPI 文档与接口代码同步维护。
- 鉴权、风控、审计和权限控制必须显式设计，不允许“先跳过后补”。
- 任何涉及交易、资产、用户身份的接口，都要明确幂等性、鉴权方式和审计要求。

## 7. 数据与数据库要求

- 表名统一使用 `aiw_` 前缀。
- 主业务实体统一使用 `biz_id` 作为业务主键。
- 默认包含 `created_at`、`updated_at`，必要时包含 `created_by`、`updated_by`、`is_deleted`。
- 数据库结构变更必须通过 Flyway 迁移脚本落地。
- 禁止数据库外键和级联更新、级联删除；跨表使用逻辑 `biz_id` 并按查询路径建立索引。
- 跨领域一致性通过应用用例、领域事件和补偿机制实现，不依赖数据库隐式级联。
- 表设计、实体设计和文档必须同一轮变更内保持一致。

## 8. 测试与质量门禁

- 领域规则必须有单元测试。
- 仓储、外部集成和关键流程必须有集成测试。
- 修改数据库结构时，至少验证应用上下文和迁移可执行性。
- PR 必须说明变更目标、影响范围、测试方式和回滚思路。
- 任何 AI 参与生成的代码都必须经过人工复核。

## 9. AI 辅助开发要求

- 允许 AI 生成样板代码、文档初稿、测试样例和重构建议。
- 禁止未经审查直接合并 AI 生成的业务规则、权限控制、风控和安全代码。
- 使用 AI 时必须提供足够上下文，包括业务域、包路径、边界和测试要求。
- 一次只让 AI 处理一个清晰问题，避免大面积无边界生成。

## 10. 应做与禁止事项

### 应做

- 先看 `docs_new` 再动代码。
- 先定义业务边界，再实现对象和接口。
- 一次只交付一个清晰的垂直切片。
- 代码、文档、迁移脚本同步提交。

### 禁止

- 不把历史示例业务代码当作新实现模板。
- 不在 Controller 中堆业务逻辑。
- 不把数据库表结构直接暴露成领域模型。
- 不把明文凭据提交进仓库。
